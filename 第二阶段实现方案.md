# 第二阶段实现方案：账号体系与作品持久化

## 目标

第二阶段的目标是把第一阶段的 MVP 骨架从“内存演示”推进到“可持续开发的业务底座”。核心是接入 PostgreSQL 持久化、完善账号体系、实现作品管理列表与详情页，并为后续真实 LLM、社区画廊、多模态任务打好数据基础。

本阶段完成后，系统应支持：

1. 用户注册、登录、退出和登录态恢复。
2. 用户、Agent 会话、生成应用、任务记录持久化到 PostgreSQL。
3. 用户可以保存、查看、编辑和删除自己的生成应用。
4. 前端具备登录/注册页面、我的作品列表、作品详情/编辑页面。
5. 后端具备清晰的数据访问层，后续可继续扩展评论、点赞、公开画廊和计费。

## 范围边界

### 本阶段包含

- SQLAlchemy 2.x 异步数据库接入。
- Alembic 数据库迁移。
- PostgreSQL 表结构落地。
- JWT 认证增强。
- 前端登录/注册页面。
- 作品 CRUD。
- 会话与作品关联。
- 基础错误处理和接口状态提示。

### 本阶段不包含

- 真实 LLM 接入。
- Docker 沙箱执行。
- 多模态图片/音乐生成。
- 社区公开画廊。
- 点赞、评论、向量推荐。
- OAuth 第三方登录。
- 生产级 RBAC 权限系统。

这些能力保留到第三阶段之后。

### 6. Repository 替换

当前文件：

```txt
backend/app/services/repository.py
```

第二阶段不再使用全局内存字典，改为依赖注入数据库 session。

新增依赖：

```py
async def get_db_session() -> AsyncIterator[AsyncSession]:
    async with async_session() as session:
        yield session
```

然后在路由中注入：

```py
db: AsyncSession = Depends(get_db_session)
```

Service 层调用 Repository：

```py
user_repository = UserRepository(db)
user = await user_repository.get_by_email(payload.email)
```

## 验收标准

### 后端

- `docker compose up postgres redis minio` 后，后端能连接 PostgreSQL。
- `alembic upgrade head` 能创建所有核心表。
- 注册用户后，`users` 表存在记录。
- 登录返回 JWT。
- `GET /api/auth/me` 能返回当前用户。
- 创建会话后，`agent_sessions` 表存在记录。
- 保存作品后，`generated_apps` 表存在记录。
- 用户 A 无法读取、修改、删除用户 B 的作品。

### 前端

- 未登录访问 `/` 会跳转 `/login`。
- 注册成功后进入工作台。
- 刷新页面后登录态能恢复。
- 生成应用后可以保存。
- `/apps` 能看到保存的作品。
- `/apps/:id` 能打开作品、编辑代码并保存。
- 删除作品后列表同步更新。

4. 新增 `/api/auth/me`。
5. 替换 apps/session repository。
6. 增加 apps update/delete 接口。
7. 前端新增登录页和路由守卫。
8. 前端新增作品列表页。
9. 前端新增作品详情页。
10. 联调完整链路。

## 风险与处理

### 1. 内存仓储切数据库时影响现有 Service

处理方式：先新增 Repository 类，不直接在 Service 里写 SQL。Service 只依赖 Repository 方法。

### 2. JWT token 对应用户不存在

处理方式：`get_current_user` 查询不到用户时返回 401，前端清理 token 并跳转登录页。

### 3. JSONB 消息结构后续演进

处理方式：消息里保留 `role`、`content`、`created_at` 基础字段，后续新增字段保持兼容。

### 4. 作品代码体积变大

处理方式：第二阶段先存在 PostgreSQL text 字段。后续如果生成完整项目，再把文件树存对象存储或独立 `generated_app_files` 表。
